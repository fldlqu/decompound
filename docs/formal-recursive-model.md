# Formal recursive model

This document specifies the mathematical object generated by `decompound` and
its encoding in `src/decompound/model.py`.

## Domains

Let:

- \(L\) be the finite set of licensed lexemes;
- \(T\) be the set of semantic types;
- \(R\) be the set of typed composition rules;
- \(\Sigma^*\) be the set of finite Unicode strings.

Each lexeme \(l \in L\) has:

- a lemma \(lemma(l)\);
- a licensed compound stem \(stem(l) \in \Sigma^*\);
- a licensed linking form \(link(l) \in \Sigma^*\), possibly empty;
- a semantic type \(type(l) \in T\).

Its modifier realization is:

\[
modifier(l) = stem(l) \mathbin{\|} link(l)
\]

where \(\|\) denotes string concatenation. The linking form is morphological
material but not an independently counted semantic component.

Each rule \(r \in R\) has the signature:

\[
r : (T_m, T_h) \rightarrow T_o
\]

plus a semantic relation label and an interpretation template. A rule is
applicable only if the modifier has type \(T_m\) and the current compound has
head/result type \(T_h\).

## Recursive construction

A compound state is \(C_k = (h, S_k)\), where \(h \in L\) is the immutable
rightmost lexical head and \(S_k\) is an ordered sequence of composition steps.
The base state is:

\[
C_1 = h
\]

with:

\[
count(C_1)=1, \qquad type(C_1)=type(h), \qquad spell(C_1)=stem(h)
\]

Given a current state \(C_k\), choose a modifier \(m_{k+1}\) and a rule
\(r_{k+1}\) such that:

\[
type(m_{k+1}) = r_{k+1}.modifier\_type
\]

and:

\[
type(C_k) = r_{k+1}.head\_type
\]

Then the next state is the binary composition:

\[
C_{k+1} = r_{k+1}(m_{k+1}, C_k)
\]

and satisfies:

\[
count(C_{k+1}) = count(C_k) + 1
\]

\[
type(C_{k+1}) = r_{k+1}.result\_type
\]

\[
spell(C_{k+1}) = modifier(m_{k+1}) \mathbin{\|} spell(C_k)
\]

Thus every extension adds exactly one semantic component on the left and keeps
the previous compound as its complete right-hand suffix, apart from the purely
orthographic change whereby the old standalone initial becomes lowercase inside
the larger noun.

## Tree and flat encoding

The abstract syntax is right-branching:

```text
r3(m3,
   r2(m2,
      r1(m1, h)))
```

The implementation stores the same tree without recursive Python objects:

```text
head = h
steps = (Step(m1, r1), Step(m2, r2), Step(m3, r3))
```

`steps` therefore runs from the modifier nearest to the head to the outermost
modifier. Semantic validation folds it from left to right. Orthographic
linearization reverses it and emits:

```text
modifier(m3) + modifier(m2) + modifier(m1) + stem(h)
```

This representation avoids Python recursion-depth limits. Contract generation
accumulates steps in a mutable list and freezes it once, giving \(O(N)\) model
construction and \(O(N)\) linearization for \(N\) components. Interactive
extension creates a new immutable tuple, while the previous `Compound` remains
valid and unchanged.

## Invariants

Every valid `Compound(head, steps)` must satisfy all of the following:

1. `component_count == 1 + len(steps)`.
2. The first rule consumes `head.semantic_type`.
3. Every later rule consumes the preceding rule's `result_type`.
4. Every step's modifier type equals its rule's `modifier_type`.
5. `semantic_type` equals the final rule result type, or the head type when the
   step sequence is empty.
6. `components()` yields outermost modifier through rightmost head.
7. Linearization yields one capitalized orthographic word.

`validation.validate` independently checks the type fold and the final
orthographic contract. Generator correctness does not replace validation:
generation constructs a candidate; validation verifies these invariants.

## Inductive preservation theorem

Assume \(C_k\) satisfies the invariants and an extension step satisfies its rule
signature. The new sequence retains every old valid edge, adds one edge whose
modifier and head types match the rule inputs, and declares the rule output as
the new result type. Therefore all structural and typing invariants hold for
\(C_{k+1}\). By induction they hold for every finite state reachable from a
valid head.

This theorem establishes preservation, not unbounded availability. The separate
closure construction proving that a next applicable step always exists is
specified by the safe-recursion step of the plan.
